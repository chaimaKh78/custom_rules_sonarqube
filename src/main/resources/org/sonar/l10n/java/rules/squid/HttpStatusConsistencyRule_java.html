HTTP Status Consistency

This rule ensures that HTTP status codes are used consistently based on the context of the response and the HTTP method used.

Why is this important?

Inconsistent use of HTTP status codes can lead to confusion for clients consuming your API and can violate the principles of REST. This rule helps maintain clarity in your API responses by ensuring that success and error status codes are used appropriately based on the context.

How to fix issues raised by this rule

  • Ensure that success status codes (e.g., 200 OK, 201 CREATED) are used only in appropriate success contexts.
  • Ensure that error status codes (e.g., 400 BAD REQUEST, 404 NOT FOUND) are used when an error occurs.
  • Avoid using both success and error status codes in different return paths within the same method.
  • When throwing exceptions, ensure that appropriate error HTTP status codes are returned.
