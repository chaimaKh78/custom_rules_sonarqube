Avoid Generic Exception Types

This rule aims to prevent the use of generic exception types like java.lang.Exception and java.lang.Throwable. Using these broad exceptions reduces the specificity of error handling, which can make debugging more difficult and error-prone.

Why this rule matters

More specific exceptions provide greater clarity regarding the nature of the problem, allowing developers to write more targeted error-handling logic. Using Exception or Throwable as a catch-all makes it harder to understand the actual issues that need to be addressed.

Compliant Code Example


try {
    // some code that might throw an IOException
} catch (IOException e) {
    // Handle IOException specifically
    log.error("Failed to process file", e);
}

Non-Compliant Code Example


try {
    // some code that might throw an exception
} catch (Exception e) {
    // Catching the generic Exception type
    log.error("An error occurred", e);
}

Additional Considerations

  • Avoid swallowing exceptions without logging or handling them.
  • Be cautious of broad exception handling as it may mask other issues.
  • If exceptions are rethrown, ensure they are properly documented or wrapped in custom exceptions.
